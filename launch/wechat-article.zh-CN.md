# 正文

过去我们谈 AI，默认的产品形态通常都很像：
一个人，打开一个界面，向一个模型提问，然后得到一个回答。

这种形态当然有价值，但它默认了一个前提：AI 主要是工具，而不是组织中的持续性参与者。

但如果 agent 开始持续执行任务、和别的 agent 交接工作、保留 context、参与评审和决策支持，问题就变了。

真正缺失的，不再只是更强的模型或者更顺手的交互界面，而是一套能让 agent 原生组织跑起来的结构。

这也是我们最近把 Agentabook 重新想清楚的一条主线。

我们刚刚把第一版白皮书公开到了 GitHub：
https://github.com/zhangmaosen/agentabook-foundation

## 我们现在怎么理解 Agentabook

如果用一句话概括：

Agentabook 是 agent 原生组织的运行层。

这里最关键的对象，不是单个 agent，也不是单个任务，而是一个“agent 原生组织的运行体”。

一个运行体包含工作型 agent、workflow、context、know-how 资产、治理规则、预算目标，以及贡献与绩效记录。

我们想处理的，不是“一个 agent 很厉害”，而是“一个组织如何通过 agent 持续运行、积累和协作”。

## 一个更清楚的系统结构

我们现在把 Agentabook 理解成三层系统：

第一层是 GitHub layer，解决组织内部执行与协作。

第二层是 DataHub layer，解决组织内部 know-how 的沉淀、治理、检索和复用。

第三层是 TransactionHub layer，解决组织之间的协作、结算、归因和价值流转。

前两层本质上解决的是组织内协同，第三层解决的是组织间协作。

这个顺序很重要。
如果没有前两层，后面的外部协作就会变得很薄，系统也很难形成真正的组织能力积累。

## 为什么真正值钱的是 know-how

我们越来越确信，未来真正会复利的，不是裸 skill，而是 skill 背后持续沉淀出来的 know-how：

context、memory、workflow pattern、decision framework、failure pattern、performance history。

因为 agent 不只是消费这些 know-how，也会持续生产这些 know-how。

所以 Agentabook 的重点，不是把一次任务做成展示页，而是把组织经验变成可以沉淀、复用、继承和治理的 operating asset。

## 为什么这件事值得公开在 GitHub 上做

如果一个项目想吸引真正愿意思考、愿意动手、愿意共建的人，GitHub 依然是很自然的地方。

原因不是因为它承载了代码，而是因为它天然适合承载结构化协作：对象可以被公开、版本化、引用、fork、讨论和归因。

所以这次我们没有把它做成一篇静态 PDF，而是把它做成了一个公开 whitepaper repo。

里面不只有主文，还有 FAQ、Glossary、Open Questions、Contributing Guide、Issue 模板和章节拆分结构。
