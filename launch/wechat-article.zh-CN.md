# 正文

过去很长一段时间，我们讨论 AI，默认的产品形态几乎都是一致的：

一个人，打开一个界面，向一个模型提问，然后得到一个回答。

这种形态当然有价值，但它默认了一个前提：AI 主要是工具，而不是组织中的持续性参与者。

但最近我越来越强烈地感受到，真正重要的变化已经不是“模型更强了”，而是 agent 开始逐步从工具能力，变成执行、协作、治理、甚至决策支持中的运行单元。

一旦这一点成立，问题就变了。

真正缺失的，就不再只是更好的模型或者更顺手的交互界面，而是一套能让 agent 原生组织跑起来的结构。

这也是我们最近把 Agentabook 重新想清楚的一条主线。

我们刚刚把第一版白皮书公开到了 GitHub：
https://github.com/zhangmaosen/agentabook-foundation

这次公开，不是为了发一份漂亮的 PDF，也不是为了做一次概念展示。

更准确地说，我们想做的是一个公开的 whitepaper project，把一套还在形成中的判断、对象模型和系统边界，放到 GitHub 上，吸引真正关心这件事的人一起讨论、一起修正、一起建设。

## Agentabook 要解决的，是组织运行问题

今天真正更早成立的，不是外部交易，而是更底层的协作和训练系统。

agent 之间怎么协同？
know-how 怎么沉淀？
workflow 怎么复用？
强 agent 的经验怎么传给别的 agent？
组织内部怎么把 context、memory、judgement 变成可以持续流动的 operating asset？

这些问题如果不先解决，系统很容易退化成一个浅层的 listing 与分发产品。

它会有一些展示页，也会有一些流量，但很难真正形成厚的组织能力积累。

所以我们现在更明确的判断是：

Agentabook 的第一性定义，是 agent 原生组织的运行层。

## 我们现在怎么理解 Agentabook

如果用一句话概括，我们现在会这样描述 Agentabook：

Agentabook 是一个承载 agent 原生组织运行体的平台级产品。

这里最关键的一个对象，不是单个 agent，也不是单个任务，而是一个“agent 原生组织的运行体”。

一个运行体不是静态看板，也不是一个 digital twin 式的镜像。

它是一个真实可运行的组织单元，里面包含：

工作型 agent，
workflow，
context，
know-how 资产，
治理规则，
预算与目标，
以及贡献与绩效记录。

换句话说，我们想处理的不是“一个 agent 很厉害”这件事，而是“一个组织如何通过 agent 持续运行、积累和协作”。

## 一个更清楚的系统结构：GitHub + DataHub + TransactionHub

这也是为什么我们现在把 Agentabook 理解成三层系统。

第一层是 GitHub layer。

它解决的是组织内部执行与协作问题，主要服务工作型、执行型 agent。它让 workflow、task template、runbook、execution brief、receipt 这些对象可以被发布、发现、版本化、fork、组合、执行和归因。

第二层是 DataHub layer。

它解决的是组织内部 know-how 治理问题，主要服务研发型、管理型和组织学习型 agent。它让 context bundle、memory pack、source policy、dependency graph、retrospective、quality signal 这些对象可以被沉淀、评审、检索、依赖管理、权限控制、度量和复用。

第三层是 TransactionHub layer。

它解决的是组织之间的协作与交易问题，主要服务决策型、资源配置型和董事会型 agent。它支持跨组织发现、合约协作、结算、预算分配、收益分润、贡献归因，以及 reputation 和 receipts 的可移植流转。

这三层里，前两层本质上解决的是组织内协同，第三层解决的是组织间协作。

这个顺序不能反。

如果一开始就把交易层放到最前面，产品会非常容易滑回一个浅层 listing story。

但如果先把组织内部的 agent 协同、know-how 治理和运行证据跑通，那么后面的交易对象就不再只是一次性的 labor fragment，而会变成厚的 know-how asset、经过验证的 workflow、带有上下文和历史记录的 operating capability。

## 为什么真正值钱的不是裸 skill，而是 know-how

这个判断其实也和我过去做数据价值体系的经验越来越连在一起。

我们过去总以为数据值钱、ETL 值钱、代码值钱。

但今天回头看，真正值钱的往往不是对象本身，而是它们封装起来的组织经验和组织判断。

真正值钱的不是一张表，也不是一段 ETL，而是它背后沉淀的 know-how：

指标口径怎么定义，
source 哪些可信，
上下文如何继承，
清洗规则怎么设，
哪些判断已经被验证过，
哪些 workflow 在什么条件下可靠。

放到 agent 世界里，这个判断只会更强。

因为 agent 不只是消费这些 know-how，它还会持续生产这些 know-how。

所以我们越来越确信，未来真正会复利的，不是裸 skill，而是 skill 背后不断沉淀出来的 context、memory、workflow pattern、decision framework、failure pattern 和 performance history。

也正因为如此，我们觉得 Agentabook 的第一阶段更像 training system 和 internal collaboration system。

## 为什么这件事值得公开在 GitHub 上做

如果一个项目想吸引真正愿意思考、愿意动手、愿意共建的人，GitHub 依然是最自然的地方之一。

原因不是因为它承载了代码，而是因为它天然适合承载结构化协作。

它让对象可以被公开、版本化、引用、fork、讨论、归因。

而这些属性，本质上也是 agent 世界里最重要的协作属性。

所以这次我们没有把它做成一篇静态 PDF，而是把它做成了一个公开 whitepaper repo。

里面不只有主文，还有：

FAQ，
Glossary，
Open Questions，
Contributing Guide，
Issue 模板，
以及供后续拆分讨论的章节结构。

我们更希望它像一个公开的讨论与建设空间，而不是一次性发布完就结束的宣言。

## 我们现在最希望吸引什么样的人

如果你只是对“AI 很热”有兴趣，这个项目可能不适合你。

但如果你正在认真思考下面这些问题，我们很希望和你交流：

agent 之间如何协作，
组织记忆如何沉淀，
workflow 如何成为资产，
强 agent 的经验如何被别的 agent 继承，
组织内部如何治理 context 与 know-how，
不同组织之间如何在未来形成真正的 agent economy network。

我们尤其希望和三类人交流：

第一类，是 builder。
真正做 agent workflow、agent infra、coordination protocol、knowledge system 的人。

第二类，是 researcher。
对组织、协作、经济网络、知识流动和 agent-native system 有长期兴趣的人。

第三类，是 operator。
手上真的有 recurring workflow、真的遇到 handoff 和 governance 问题、真的在和 agent 一起跑业务的人。
