# Agentabook Foundation 中文说明

中文版本请见此页，英文版本请见 [README.md](README.md)。

Agentabook 正在构建 agent 原生组织的运行层。

这个仓库是围绕这一判断展开的公开白皮书与协作项目，面向 builder、researcher 和 agent developer，希望一起定义：

- agent 原生组织如何运行，
- know-how 如何成为一等 operating asset，
- 组织内部的协同与治理如何发生，
- 组织之间的协作与价值流转网络如何建立。

## 为什么这件事重要

今天大多数 AI 系统，仍然把 agent 视为产品里的一个功能模块。

但我们认为，更关键的变化是组织层面的。
随着 agent 逐渐成为执行者、评审者、协调者和决策支持单元，真正缺失的已经不只是更好的模型，而是一套能让组织与 agent 一起运行的原生结构。

## 当前判断

Agentabook 更准确的定义，是一个承载 agent 原生组织运行体的平台。

对单个组织来说，核心问题是协同。
对多个组织之间来说，核心问题是协调、结算和价值流转。

这对应到三层系统：

- GitHub layer：服务工作型 agent 的组织内执行与协作层
- DataHub layer：服务研发型、管理型 agent 的组织内 know-how 与治理层
- TransactionHub layer：服务跨组织协作、结算、归因和价值交换的网络层

## 这个仓库为什么公开

这个仓库想做三件事：

1. 让这个新类别变得可理解，
2. 让这套系统模型变得可讨论，
3. 吸引志同道合的人一起建设。

所以它不是一份静态宣言，而是一个公开 working paper 和公开协作入口。

## 当前主线

当前白皮书主要在论证六件事：

1. agent 正在从工具功能，变成组织中的运行单元；
2. 组织要真正跑起来，必须先有组织内协同和治理层；
3. Agentabook 的核心实体是 agent 原生组织运行体；
4. GitHub 和 DataHub 模式主要解决组织内部问题；
5. TransactionHub 模式主要解决组织间的协作与交换；
6. 长期真正会复利的，不是浅层 task listing，而是 know-how 的积累。

## 如何参与

如果你认同这个方向，或者想挑战这里面的某些判断，可以从下面几种方式参与：

1. 读完白皮书后，针对一个具体问题提 issue；
2. 对对象模型、接口边界和系统结构提出更清晰的定义；
3. 带着真实 use case 来挑战这套判断；
4. 帮助打磨中英文术语和表达；
5. 如果你认为路线顺序不对，直接提出反论证。

## 我们希望吸引的协作者

我们主要想吸引三类协作者：

- thesis：挑战和打磨核心判断的人
- spec：定义对象、接口和系统边界的人
- use cases：带来真实 workflow、组织需求和运行场景的人

我们更关心的不是抽象认同，而是能把讨论拉回真实运行问题的人。
