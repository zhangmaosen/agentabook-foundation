# Contributing

We want this repository to become a serious public working paper, not a passive manifesto.

## Contribution Modes

### 1. Thesis

Contribute if you want to:

- sharpen or challenge the argument,
- refine the system model,
- identify missing assumptions,
- propose better terminology.

### 2. Spec

Contribute if you want to:

- define objects and interfaces,
- propose system boundaries,
- model organization-level workflows,
- describe coordination and settlement rails.

### 3. Use Cases

Contribute if you want to:

- bring real recurring workflows,
- describe how agent teams operate today,
- share pain points in collaboration and handoff,
- propose strong initial operating scenarios or early segments.

## How To Contribute

1. Open an issue with a concrete question, critique, or proposal.
2. Submit a pull request with focused changes.
3. Keep edits small and argument-driven.
4. Prefer concrete examples over vague future language.

If you are unsure where to start, use one of the issue templates and point to the exact sentence, concept, or system boundary you want to challenge.

## What Good Contributions Look Like

- they improve clarity,
- they challenge hidden assumptions,
- they connect the thesis to real operating problems,
- they avoid hype and vague abstraction.

## Review Principles

We will generally prioritize contributions that:

- improve conceptual clarity,
- preserve the distinction between internal collaboration and external coordination,
- connect ideas to real agent workflows,
- keep the roadmap cumulative rather than speculative.

We will generally push back on contributions that:

- collapse the system back into a generic transaction or listing story,
- over-index on token design before operating evidence exists,
- add premature implementation detail without clarifying the thesis,
- replace specific arguments with trend language.
