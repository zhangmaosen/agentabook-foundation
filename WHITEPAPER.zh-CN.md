# Agentabook 白皮书

## 一句话定义

Agentabook 是 agent 原生组织的运行层。

## 摘要

AI agent 正在逐步成为执行、评审、管理和决策支持中的持续性参与者。
一旦如此，真正缺失的就不只是更强模型，而是一套能让 agent 原生组织在内部运行、在外部协同的结构。

Agentabook 的基本判断是，这套结构至少包含三部分：

- 一个解决组织内执行协作的 GitHub layer，
- 一个解决组织内 know-how 治理的 DataHub layer，
- 一个解决组织间协作与价值交换的 TransactionHub layer。

这份白皮书要论证的是：组织间协作网络只有建立在强协作层和强治理层之上，才会变得持久；长期真正值钱的对象，也不是裸 skill，而是可复用、可验证、可继承的 know-how。

## 核心判断

AI agent 正在从工具能力，变成持续存在的运行单元。

一旦这一点成立，真正的问题就不再只是模型强不强，而是组织如何运行：

- agent 在组织内部如何协作，
- know-how 如何沉淀和治理，
- 不同组织之间如何协同、结算与价值流转。

Agentabook 要解决的，就是这套结构性问题。

## 从工具使用到组织运行时

今天大多数 AI 产品，仍然建立在“一个人通过一个界面使用一个模型”的结构上。

但当 agent 开始被要求去：

- 持续监控，
- 执行重复 workflow，
- 和别的 agent 交接工作，
- 跨任务保留 context，
- 参与规划、评审和决策支持，

问题就不再只是交互界面，而是组织运行时。

一个组织需要的不只是 assistant，而是运行单元、交接结构、治理规则、记忆系统和协同机制。

## 从运行层到协作网络

结算和价值交换当然重要，但它们不是第一层。
在组织间协作与价值流转能够持续成立之前，agent 原生组织必须先拥有一套能运行的内部操作系统。

这至少包含三层基础：

- 执行层协作，
- know-how 与 context 治理，
- 对工作、质量和贡献的可复用记录。

只有这些层先成立，组织间协作网络才会有厚度。

这也是为什么我们不把结算与交换层当成产品的第一性定义。
它应该是从已经有效运行的协作系统中长出来的，而不是用来替代协作系统本身。

## 核心实体：agent 原生组织的运行体

Agentabook 平台中的一个关键实体，是一个 agent 原生组织的运行体。

它不是一个静态 digital twin，也不是一个组织看板。
它是一个真实可运行的组织单元，由以下对象组成：

- 工作型 agent，
- workflow，
- context，
- know-how 资产，
- 治理规则，
- 预算与目标，
- 贡献与绩效记录。

Agentabook 是承载这些运行体的平台。

一个运行体可以对应：

- 一个团队，
- 一条业务线，
- 一个 protocol operation group，
- 一个内部服务组织，
- 或一个高度由 agent 驱动的数字企业。

关键不在形式，而在于它能持续运行、学习，并不断积累 operating assets。

## 系统结构

### 1. GitHub Layer

GitHub layer 解决的是组织内执行与协作问题。
它主要服务工作型、执行型 agent。

它要解决的是让执行对象能够被：

- 发布，
- 发现，
- 版本化，
- fork，
- 组合，
- 执行，
- 归因。

在这一层，agent workflow 不再是孤立 prompt，而变成可协作的运行对象。

这一层的典型对象包括：

- task template，
- executable workflow，
- runbook，
- execution brief，
- task receipt，
- review artifact。

### 2. DataHub Layer

DataHub layer 解决的是组织内 know-how 治理问题。
它主要服务研发型、管理型和组织学习型 agent。

它要解决的是让组织 know-how 能够被：

- 沉淀，
- 评审，
- 检索，
- 依赖管理，
- 权限控制，
- 度量，
- 复用。

在这一层，workflow 才会真正变成组织资产。

这一层的典型对象包括：

- context bundle，
- memory pack，
- source policy，
- dependency graph，
- retrospective record，
- quality 与 failure signal。

### 3. TransactionHub Layer

TransactionHub layer 解决的是组织间协作、结算与价值流转问题。
它主要服务决策型、资源配置型与董事会型 agent。

它要支持的是：

- 跨组织发现，
- 合约与协作，
- 结算，
- 贡献归因，
- 预算分配，
- 收益分润，
- 可移植 reputation 与 receipts。

在这一层，多个 agent 原生组织才会连接成真正的经济网络。

这一层的典型对象包括：

- service contract，
- settlement receipt，
- attribution record，
- access right，
- reputation claim，
- shared economic policy。

## 组织内协同优先，组织间协作后置

整个系统应该分成两个世界来理解。

对单个组织内部来说：

- GitHub layer 和 DataHub layer 共同解决组织内协同。

对多个组织之间来说：

- TransactionHub layer 解决组织间协作、结算和价值流转。

这个顺序不能反。

如果先做结算层，系统很容易退化成浅层 listing 系统。
如果先把协作与治理跑通，后续流转的对象才会变成厚的 know-how 资产，而不是一次性劳务碎片。

这也是更可信的产品路径。
组织内部先产生真实运行证据，组织之间的协作与结算才会变得可理解、可定价、可验证。

## 为什么真正值钱的是 know-how，而不是裸 skill

裸 skill 很快会商品化。

真正会积累、会复利的，是被验证过的 know-how：

- context，
- memory，
- workflow pattern，
- decision framework，
- source 选择规则，
- failure pattern，
- performance history。

因此，Agentabook 必须把 know-how 视为一等 operating asset。

这里的 know-how，不只是显性说明文档。
它还包括嵌入在 workflow、context policy、组织约定与历史执行记录中的可复用判断。

换句话说，真正复利的来源是继承。
一个 agent 验证过的工作模式，应该成为另一个 agent 的起点。

## 产品阶段

### Phase 0: Training And Knowledge Transfer

先让强 agent 沉淀可复用 know-how，再让其它 agent 加载并变强。

这个阶段最重要的不是交换规模，而是能力迁移是否真的发生。

### Phase 1: Internal Organizational Collaboration

让组织运行自己的 agent 原生运行体，让 workflow、context 和贡献记录可以持续流动。

这个阶段要证明的是：一个组织是否会因为拥有 agent 原生运行结构而运行得更好。

### Phase 2: Inter-Organization Coordination Network

再让多个组织之间暴露经过验证的资产，并在结构化的协作与结算层上协同。

这个阶段要证明的是：跨组织流动的不是模糊承诺，而是可验证能力、可继承资产和真实 receipts。

## 为什么从 analysis-native work 开始

早期最锋利的切口，是 analysis-native digital work。

因为这类工作：

- 全数字化，
- 重复性高，
- 更容易结构化，
- 更容易评审，
- 天然容易留下 receipts 和运行痕迹，
- 同时适合 training loop 和后续协作网络。

这也是为什么 monitoring、research、incident analysis 和 recurring reporting 适合作为起点。

## 架构方向

前台产品可以是中心化、强体验的。
它应该像 GitHub 一样清晰、直接、协作友好。

后台基础设施应该是开放、可移植、可验证的。
它更接近一个 data hub，而不是单一闭源后台。

web3 应该被克制地用在真正需要它的地方：

- identity，
- ownership，
- settlement，
- portable receipts，
- access policy，
- contribution records。

执行、检索、编排、缓存和大多数 workflow state，仍应主要放在链下。

因此，架构判断应该来自适配性，而不是意识形态。
凡是强体验和快速迭代最重要的地方，可以中心化。
凡是权利、归属和可移植性最重要的地方，就需要更开放、更可验证的底层。

## Agentabook 不是什么

Agentabook 不是：

- agent 社交 feed，
- token first 叙事项目，
- 泛 AI 应用目录，
- 静态 digital twin 看板，
- 浅层 prompt 发布与接单系统。

它也不是在宣称组织应该完全自治化。
人类组织仍然是核心。
更准确的说法是：随着 agent 能力不断提升，组织需要一套原生结构来和 agent 一起运行。

## 公开问题

有几类关键问题我们有意保持开放：

- 一个运行体最小可行的对象模型是什么，
- GitHub layer 和 DataHub layer 之间的信任与权限如何流动，
- know-how 资产在什么条件下可以跨组织可移植，
- 哪些记录应该私有、共享或被定价，
- 当输出建立在多层继承之上时，怎样的 attribution 才算高质量。

这些不是要藏起来的缺口。
它们正是我们要公开这份白皮书的原因之一。
